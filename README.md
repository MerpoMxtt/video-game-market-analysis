# Video Game Market Analysis

An end-to-end data project built across three layers — data engineering, 
data analysis, and data science — using a large-scale simulated video game 
market dataset.

Built as a portfolio project to demonstrate real-world skills in database 
design, SQL analytics, and machine learning.

---

## What This Project Does

This project takes a raw 5GB dataset of video game market data and builds 
a full pipeline from scratch:

1. **Engineering layer** — loads raw data into a structured database
2. **Analysis layer** — runs SQL queries to answer business questions
3. **Data science layer** — trains a machine learning model to predict 
   breakout hit games

Each layer feeds the next, which mirrors how real data teams are structured.

## How to Reproduce This Project

### 1. Clone the repo
```bash
git clone https://github.com/MerpoMxtt/video-game-market-analysis.git
cd video-game-market-analysis
```

### 2. Create and activate a virtual environment
```bash
python -m venv .venv
.venv\Scripts\activate        # Windows
source .venv/bin/activate     # Mac/Linux
```

### 3. Install dependencies
```bash
pip install -r requirements.txt
```

### 4. Download the dataset
```bash
python download_data.py
```
This downloads ~5GB from Kaggle into your local cache.
Requires a free Kaggle account.

### 5. Build the database
```bash
python engineering/ingest.py
```
This scans the raw CSV and builds a structured DuckDB database.
Takes a few minutes.

### 6. Verify it worked
```bash
python engineering/verify_db.py
```

### 7. Open the notebooks
Open `analysis/exploratory_analysis.ipynb` and 
`modeling/breakout_classifier.ipynb` in VS Code or Jupyter, 
select the `.venv` kernel, and run all cells.

## Layer 1 — Data Engineering

**File:** `engineering/ingest.py`

The raw dataset is a single flat CSV with 130 columns and 5.58 million rows. 
Keeping it flat means every static attribute (developer, genre, release date) 
gets repeated 558 times per game — once per weekly observation. That's wasteful 
and slow to query.

The solution is a **star schema** — a standard pattern in analytics databases 
where you separate descriptive information (dimensions) from measurements (facts).

**Tables built:**

| Table | Rows | What it contains |
|---|---|---|
| `dim_games` | 9,995 | One row per game — name, developer, genre, release date, etc. |
| `dim_platform` | 6 | One row per platform — PlayStation 5, PC, Nintendo Switch, etc. |
| `dim_time` | 2,190 | One row per date — year, month, quarter, season flags |
| `fact_weekly_metrics` | 5,584,706 | One row per game × platform × week — price, players, revenue, targets |

**Why DuckDB instead of pandas?**

Pandas loads the entire file into RAM. At 5GB that's slow and potentially 
crashes on a laptop. DuckDB queries the file on disk and only pulls what 
it needs — much faster for large analytical queries.

## Layer 2 — Data Analysis

**File:** `analysis/exploratory_analysis.ipynb`

Four business questions answered using SQL queries against the DuckDB database:

**1. Which genres generate the most revenue?**
First-Person Shooter ($81M avg/week) and RPG ($79.6M) lead.
Indie and Visual Novel sit at the bottom (~$20M).

**2. How do players and revenue change across a game's lifecycle?**
The most interesting finding — player counts decay over time 
(110k at launch → 17k at long tail) but **revenue actually increases** 
in later phases. The reason: discount depth grows from 0.45% at launch 
to 7.74% in the long tail, driving more total purchases even with fewer 
concurrent players. This is the "discount-driven long tail revival" effect.

**3. Which platform performs best?**
PlayStation 5 leads slightly (35k avg players, $61M avg revenue) but 
all six platforms are remarkably close — a likely artifact of the simulation 
distributing games evenly across platforms.

**4. Do discounts drive player spikes?**
The 1–9% discount bucket has the highest average players (91k), 
but this is misleading — small discounts coincide with the launch phase 
when organic player counts are naturally highest. Heavier discounts 
(50–74%) show significantly lower player counts, suggesting deep discounting 
is a late-lifecycle tool, not a growth driver.

## Layer 3 — Data Science

**File:** `modeling/breakout_classifier.ipynb`

**Goal:** Predict whether a game will become a "breakout hit" based on 
its current observable characteristics.

**Target:** `target_breakout_hit` (binary: 1 = breakout, 0 = not)  
**Class balance:** 86.7% not breakout / 13.3% breakout (imbalanced)

### Models trained

| Model | ROC-AUC | Notes |
|---|---|---|
| Logistic Regression (baseline) | 0.8562 | Simple, interpretable |
| XGBoost v1 | ~0.99 | High but suspicious |
| XGBoost v2 (final) | 0.9890 | Honest model after leakage fix |

### Key finding — feature leakage

The first XGBoost model assigned 86% of its importance to one feature: 
`is_aaa`. The model wasn't learning anything meaningful — it had discovered 
that in this simulation, AAA games are simply defined as breakout hits. 
That's **feature leakage**: a feature that encodes the answer rather than 
predicts it.

Removing `is_aaa` and retraining produced a more honest model with a 
genuinely better ROC-AUC (0.989) and interpretable feature importance:

| Feature | Importance | What it means |
|---|---|---|
| `publisher_avg_budget` | 0.40 | Big marketing spend drives visibility |
| `is_multiplayer` | 0.10 | Multiplayer games have built-in virality |
| `concurrent_players` | 0.08 | Early traction predicts future traction |
| `hype_score` | 0.07 | Pre-release momentum matters |
| `twitch_viewers_proxy` | 0.06 | Streamer pickup drives breakout growth |

**The business insight:** Breakout hits are largely structurally determined 
by publisher resources and multiplayer design — not by luck or genre. 
Early engagement signals (hype, Twitch views) are the best real-time 
indicators once a game is live.

## Key Learnings

- **Star schema design** — separating facts from dimensions is standard 
  practice in analytics warehouses (Snowflake, BigQuery, Redshift all 
  use this pattern)
- **OLAP vs OLTP** — DuckDB is built for analytical queries across millions 
  of rows; pandas is built for smaller in-memory work
- **Feature leakage** — a model that looks too good usually is; always 
  inspect feature importance before trusting results
- **Class imbalance** — with 87/13 split, raw accuracy is misleading; 
  ROC-AUC and recall on the minority class are better metrics
- **Correlation vs causation** — the 1–9% discount bucket looked like 
  the best performer until we realized it overlaps with the launch phase
